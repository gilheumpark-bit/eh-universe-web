## 1. 핵심 철학 — 해줘 vs 해라 (카테고리 선언)

> **"해줘는 컨텐츠 소비. 하라는 제작이다."**
> — Loreguard 카테고리 선언 (2026)

이 한 줄이 Loreguard의 모든 결정의 기준점이다.
기능 설계, 가격 정책, 마케팅 메시지, 특허 전략, 경쟁사 분석 모두 이 선언에서 파생된다.

**두 시장이 있다. 같은 시장에서 경쟁하지 않는다.** 자세한 이론은 Part 13 참조.

### 1.1 주어가 다르다

| 구분 | 주어 | 행위자 | AI의 위치 |
|------|------|--------|-----------|
| 해줘 | AI | AI가 쓴다 | **작가의 자리** |
| 해라 | 작가 | 작가가 AI로 쓴다 | **작가의 도구** |

경쟁사: **"AI가 써줍니다"**
우리: **"작가가 AI로 씁니다"**

### 1.2 권리 관점

- **위임(해줘)** = 창작 권리의 포기. 결과물의 소유감·성장·정체성을 동시에 내주는 것.
- **통제(해라)** = 창작 권리의 유지. 도구는 강력해지되 주어는 항상 사람.

### 1.3 팀 룰 — "AI가 대신 써준다" 금지 리스트

우리 마케팅·카피·UI에서 **영구 금지** 표현:
- "AI가 대신 써드려요"
- "자동 집필"
- "AI한테 맡기세요"
- "AI가 생성한 소설"

허용 표현:
- "AI와 같이 쓰세요"
- "AI로 씁니다"
- "AI가 제안, 작가가 결정"
- "질문하면 답이 옵니다"

### 1.4 전략적 트레이드오프 (의식적 선택)

이 선언으로 **우리는 다음 시장을 포기한다**:
- Sudowrite / NovelAI / Novelcrafter의 "AI writes FOR you" 유저층
- "글 쓰기 귀찮은데 대신 써줬으면" 니즈의 일부

대신 **우리가 얻는 것**:
- 작가 정체성을 지키려는 장기 충성 사용자
- 작가 커뮤니티·출판사·평론계 지지
- "AI로 썼으니 진짜 작가 아니다"는 프레임 차단

**이 트레이드오프는 의도된 것**. 향후 "왜 이 기능 없어요?" 질문에 흔들리지 말 것.

## 2. 제3의 길 — 같이 하세요

### 2.1 3가지 포지션

| 입장 | 주장 | 한계 |
|------|------|------|
| AI 반대 | "AI 쓰지 마세요" | 도구 거부 = 시대 거부 |
| AI 위임 | "AI한테 맡기세요" | 정체성·권리 포기 |
| **제3의 길** | **"같이 하세요"** | **공생. 둘 다 살아남음** |

### 2.2 UX 번역 — 언어 가이드

버튼·기능명·에러 메시지까지 이 원칙을 적용:

| 금지 | 허용 |
|------|------|
| "자동 작성" | "함께 쓰기" |
| "AI가 완성" | "같이 고치기" |
| "AI 답변" | "질문해서 얻기" |
| "AI 생성" | "AI 제안" |
| "대신 써드립니다" | "같이 써봐요" |

### 2.3 Pull vs Push UX 원칙

**기본 = Pull (작가가 질문할 때만 깊이 도움)**
- AI 제안 폭격 = 작가 플로우 파괴
- 침묵은 존중이다

**단, Push 모드는 옵션으로 제공**
- 웹소설 작가 현실: "속도가 생명" 니즈 존재
- "제안 자동 띄우기 On" 토글 = 사용자가 스스로 선택한 통제
- 기본값은 Off, 작가가 명시적으로 켤 때만 작동

## 3. 미래형 문해력 — 스토리텔링 = 프롬프트 엔지니어링

### 3.1 메타 관점

AI 시대에 가장 중요한 스킬은 **질문을 잘 하는 능력**.
질문을 잘 하는 훈련 = **스토리텔링 훈련**.

- 상황 설정 → Context
- 캐릭터 욕구 → Goal
- 장애물 → Constraint
- 결말 구조 → Expected Output

**서사 구조를 아는 사람이 AI를 가장 잘 쓴다.**

### 3.2 포지셔닝 확장

로어가드는 단순 글쓰기 툴이 아니다:
- 1차: AI 집필 스튜디오
- 2차: **AI 시대 문해력 훈련 플랫폼**

브릿G·문피아에서 소설 잘 쓰는 법을 배우는 것 = AI에게 잘 질문하는 법을 배우는 것.
이 훈련이 된 사람은 자연스럽게 바이브 코딩·에이전트 시대의 감각을 얻는다.

### 3.3 교육 마케팅 활용

- "소설 쓰기는 프롬프트 엔지니어링의 기초다" ← 일반 대중
- "스토리텔링 훈련 = 차세대 문해력" ← 학부모·교육계
- "코드 안 배워도 AI 시대에 살아남는 법" ← 비개발자 청중

## 4. AI 통제 철학

### 4.1 레벨 5 vs 6

| 레벨 | 모델 행동 | 사용자 체감 |
|------|-----------|-------------|
| 5 (통제) | 사용자 지시에만 응답 | "내 도구" |
| 6 (자율) | 모델이 먼저 제안·판단 | "내가 도구" |

**자율이 진보가 아니라 퇴보일 수 있다.**
"더 똑똑한 AI = 더 자율적 AI"는 기술 진보 서사일 뿐, 사용자 권력 관점에선 역행이다.

### 4.2 표현 — 청중별 구분

| 청중 | 표현 |
|------|------|
| B2B·기술·투자자 | "AI 통제의 끝은 자율이 아니라 완전한 통제" |
| 일반 작가 | "AI의 진화 = 사용자에게 더 많은 통제권을 돌려주는 것" |

### 4.3 UX 번역

- AI가 먼저 튀어나오는 모달·알림 최소화
- 작가가 **Ctrl+Space / Tab / 질문 버튼**을 눌렀을 때만 깊이 응답
- 품질 게이트 자동 재작성은 "사용자가 명시적으로 활성화한 경우에만"

## 5. 유니버스 우선 · 집필 먼저 (UX 이원화)

### 5.1 내부 엔진 원칙

**유니버스(세계관)가 먼저, 그다음 소설.**
- 세계관·캐릭터·룰이 정리될수록 AI 출력 품질 보장 가능
- 설정 데이터가 없으면 ANS 10.0 엔진의 QFR/CRL/HPP 규칙이 무용지물

### 5.2 표면 UX 원칙

**집필이 먼저 보이게.**
- 웹소설 작가 대다수는 "일단 써야 아이디어가 나옴"
- 첫 화면에서 "세계관 설정부터 하세요" 강제하면 이탈

### 5.3 실제 구현 — 점진적 유도

| 진행 단계 | AI 행동 |
|-----------|---------|
| 첫 3화 | 세계관 없이도 풀 퀄리티 제공 (품질 책임 유지) |
| 4화 | "이 작품 세계관을 정리하면 일관성 +30% 보장" 안내 |
| 5화+ | 세계관 미등록 시에만 품질 저하 경고 (강제 X) |

### 5.4 외부 카피

내부 철학: "설정 없이 집필 = 품질 책임 안 짐" (진심)
외부 카피: **"설정을 먼저 잡을수록, 품질과 일관성에 우리가 더 책임질 수 있다"** (순화)

## 6. 저작권 · 사용자 데이터 (업계 표준 약관)

### 6.1 기본 원칙

- 사용자 창작물의 저작권은 **사용자에게 귀속**
- 서비스 운영에 필요한 범위에서 **비독점 사용권** 부여 (업계 표준)
- 세부 조건은 `/terms` 페이지(Part 3 IP & License) 및 `/privacy` 페이지 참조
- 약관 변경 시 **사전 공지** 의무

**카테고리 연결**: 하라 시장(제작) 사용자에게 저작권은 **수익과 직결**된다. 연재료 · 인세 · 판권 수익이 여기서 파생. 해줘 시장(소비)과 달리 약관 불투명성은 즉각 이탈 사유. 참조: Part 13.4.

### 6.2 유연성 확보

- 구체적 조항은 법무 검토 후 **약관 문서에서 관리** (브랜드 철학 문서에 하드코딩 금지)
- 향후 제품 발전·시장 변화에 맞춰 조정 가능해야 함
- **브랜드 철학 = 장기 방향성**, **약관 = 실행 세부 조건** — 두 층 분리

### 6.3 세계관 원본 라이선스

- EH Universe 공식 세계관·캐릭터·설정 원본: **CC-BY-NC-4.0**
- 상업적 2차 활용 시 별도 협의

## 7. 슬로건 · 태그라인 · 피칭 문장

### 7.1 회사 슬로건

> **"같이 하세요."**

- 경쟁사가 흉내내기 어려운 톤 (Sudowrite의 "Write Better, Faster" 같은 생산성 언어가 아님)
- 한국어 특유의 의존·공동 뉘앙스 = 해외 번역 시에도 "Write Together" 혹은 "Write With, Not For"로 확장 가능

### 7.2 제품 태그라인

> **"쓰지 말라는 아닙니다. 그렇다고 맡기라는 뜻도 아닙니다."**

- 풀 문장 버전 (긴 카피용)
- 단축형: **"쓰지 마라도, 맡기라도 아닙니다"**

### 7.3 피칭 한 줄 (컨퍼런스·슬라이드 1장)

> **"해줘 vs 해라. 주어가 다릅니다."**

투자·발표·데모 **무조건 이 문장부터**. 나머지 설명은 이 한 줄의 해설.

### 7.4 비전

> **"2,000만 예비 작가를 진짜 작가로."**

**⚠️ 숫자 근거 필수 확보** (피치 실사 단계 필수):
- [ ] 국내: 노벨피아(약 80만)·문피아(약 50만)·카카오페이지·조아라 등록 작가 합산
- [ ] 광의: 글쓰기 플랫폼 미등록이나 창작 활동 중인 인구 (OECD·통계청 근거)
- [ ] 근거 미확보 시 대체: **"국내 대형 플랫폼 등록 작가 200만+, 잠재층 1,000만+"** (검증 가능한 수치로)

### 7.5 기술 철학 (B2B·개발자용)

> **"AI 통제의 끝은 자율이 아니라 완전한 통제."**

일반 청중엔 디스토피아처럼 읽힘. **엔지니어·투자자 대상 전용**.

### 7.6 투자 피칭 훅

> **"엔진 하나, 세계관 하나, 시장 무한대."**

~~특허 하나~~ → 특허 출원 공식 완료 시 복귀. 현재는 "엔진 하나"로 시작.

## 8. 팀 룰 — 철학 이탈 방지

### 8.1 제품 기획 체크리스트

신규 기능 추가 시 다음 질문 **전부** 통과해야 머지:

1. [ ] 이 기능의 주어는 **작가**인가? (AI가 아닌가?)
2. [ ] 이 기능은 **Pull**인가? (Push라면 기본 Off + 명시적 On 옵션인가?)
3. [ ] 이 기능 설명에 "자동/대신/맡기" 표현이 있는가? (있으면 리라이트)
4. [ ] 이 기능이 작가의 저작권·정체성·통제권을 침해하는가?
5. [ ] 이 기능이 **유니버스 데이터를 생성·활용**하는가? (엔진 품질 기여도)
6. [ ] **(v1.3 신규)** 이 기능이 **"하라(제작) 사용자"를 더 강하게** 만드는가, 아니면 **"해줘(소비) 사용자"를 끌어들이는가**? 후자면 **머지 보류**. (Part 13.5)
7. [ ] **(v1.3 신규)** 애매하면 **"전문가 작가(연재자·인세 수령자)가 이걸 원하는가?"** 기준으로 판단. No → 카테고리 이탈.

### 8.2 카피·마케팅 룰

- 금지 단어 리스트(1.3) 준수
- 허용 단어 리스트(2.2) 우선 사용

## 11. 품질 프레임 재정의 — "AI 원시 출력 ≠ 제품 품질"

### 11.1 측정 축의 분기

경쟁사와 우리는 **"품질"이란 단어를 다른 뜻으로 씀**.

| 포지션 | 품질의 정의 | 측정 대상 | 측정 지표 예시 |
|--------|-------------|-----------|----------------|
| 해줘 (Sudowrite, NovelAI) | **AI 원시 출력 = 완성품** | 모델 파라미터 · 벤치마크 점수 | GPT-4o vs Qwen 9B |
| 해라 (로어가드) | **작가 × AI 협업 결과 = 완성품** | 협업 속도 · 작가 보존도 · 완결률 | 에피소드당 작가 손질 시간 |

**품질 책임 주체가 다르면 측정 축도 달라져야 함.**
경쟁사 프레임(원시 출력 품질)으로 우리를 판단하면 자동 패배.

### 11.2 모델 크기의 역설

같이하세요 맥락에서 **큰 모델 = 좋은 모델**이 **아니다**.

| 모델 크기 | 작가 경험 | 평가 |
|-----------|-----------|------|
| 거대 (GPT-4o, Claude Opus) | **너무 잘 씀** → 작가 = 편집자. 목소리 증발. | ❌ |
| 중급 (Qwen 9B, 우리) | **씨앗 제공 + 방향 제시** → 작가가 살 붙임. 목소리 보존. | ✅ |
| 소형 (7B 이하) | 씨앗도 못 만듦 → 짜증 유발 | ❌ |

### 11.3 현장 증거

웹소설 작가 피드백에서 반복되는 말:
- "Sudowrite는 내 문장이 안 나와"
- "챗GPT로 쓰면 내 글이 아닌 것 같아"
- "완성도는 좋은데 내 거 같지 않음"

→ **너무 잘 쓰는 AI의 부작용**. 우리 9B 모델은 이 함정을 **구조적으로 회피**.

### 11.4 팀 룰 적용

- **피치덱에서 "우리 모델 더 좋음" 논리 금지**. 경쟁사 프레임 받는 것.
- 대신 **"작가 목소리 보존"**으로 언어 통일.
- 내부 품질 지표: 원시 벤치마크 대신 **작가 재작성률**, **완결률**, **작가 잔존률**.

### 11.5 한 줄 정리

> **"품질은 AI 혼자 다 할 때나 따지는 것.  
> 우리는 작가와 같이 하니까, 품질은 작가 몫."**

## 13. 카테고리 이론 (2026 신규 · v1.3)

> **"해줘는 컨텐츠 소비. 하라는 제작이다."**
> 이 선언이 Part 1~12의 암묵 전제를 **명시화**한 것.

### 13.1 시장 분할

AI 소설 도구 시장은 **두 카테고리**로 나뉜다. 같은 도구 범주처럼 보이지만 **본질적으로 다른 시장**.

| 축 | 해줘 시장 (소비) | 하라 시장 (제작) |
|----|----------------|-----------------|
| 본질 | AI가 써준 것을 **읽음** | 작가가 **씀**, AI는 도구 |
| 산업 위치 | Spotify · Netflix 계열 (엔터 소비) | Logic Pro · Adobe · VS Code 계열 (제작 도구) |
| 경쟁 | Sudowrite · Novelcrafter · AI Novelist | **Loreguard (유일)** |
| 사용자 | 작가 흉내 내는 독자 | **본업 작가, 연재자** |
| 특허 전략 | 불필요 (통제가 본질 아님) | **필수 (제어가 본질)** |
| 가격대 | $10~30 엔터 구독 | **$20~50 제작 도구** |
| 지표 | MAU · 세션 시간 | **완결률 · 인세 · 잔존률** |

### 13.2 역사적 선례 — 동일 도메인, 두 카테고리 공존

같은 산업 안에서 **소비 카테고리와 제작 카테고리가 평화롭게 공존**한 사례:

| 도메인 | 소비 카테고리 | 제작 카테고리 |
|--------|--------------|---------------|
| 이미지 | Canva | Adobe Photoshop |
| 코드/웹 | Wix · Squarespace | VS Code · JetBrains |
| 도면 | SketchUp | AutoCAD |
| 음악 | GarageBand | Logic Pro · Pro Tools |
| 영상 | CapCut | Premiere · DaVinci Resolve |

**교훈**: 양쪽 다 살아남음. 서로 **경쟁 안 함**. 사용자 · 가격 · 특허 · 마케팅 모두 **다른 시장**.

소설 쓰기 도메인에서 이 분할이 이제 **본격 시작**되는 단계. Loreguard는 **하라 카테고리 선점 기회**를 잡는다.

- ~~Sudowrite / Novelcrafter와 기능 대비 비교~~ (경쟁사 프레임 = 자동 패배)

- ~~"속도 · 생산성 · 자동화" 1차 메시지~~ (소비 카테고리 프레임)

### 13.4 가격 정당성

**하라 시장 사용자 = 도구로 수익 발생**:
- 연재료 (노벨피아 · 문피아 · 카카오페이지)
- 출간 인세
- 웹소설 플랫폼 수익 공유
- 번역 판권 (JP · CN · EN)

→ 월 수익 $500~5,000 구간의 연재 작가에게 **월 $20~50 도구값 = 수익의 1~5%**.
**저비용 · 필수 지출** 구간. Adobe · JetBrains 결제 감각과 동일.

vs **해줘 시장 사용자 = 취미/실험**:
- 결제 의사 상한 $10~30
- 엔터 구독 감각 (Netflix · Spotify 가격대)
- 가격 저항 즉시 이탈

**결론**:
- 가격은 **하라 시장 기준**으로 설정. $20~50 Pro 티어가 저가 아님.
- 해줘 시장 가격 대비 비싸 보이는 건 **카테고리가 다르기 때문**. 사과할 이유 없음.

### 13.5 팀 체크리스트 업데이트 (Part 8.1 보강)

   - 해줘 유입 → ❌ **머지 보류. 카테고리 이탈.**
