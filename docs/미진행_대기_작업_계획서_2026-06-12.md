작성일: 2026-06-12  
대상: EH 부모 폴더 기준 전체 작업 정리 / `eh-universe-web` 신버전 반영 후보  
성격: 논의되었으나 아직 구현·정리·삭제가 완료되지 않은 항목의 실행 계획

---

## 0. 기준

이 문서는 사업계획서가 아니라 실행 대기 목록이다.

현재 방향은 다음으로 고정한다.

- Loreguard는 AI 소설 생성앱이 아니라 창작 전문 IDE다.
- 핵심 상품은 결과 생성량이 아니라 창작 과정기록, 승인 로그, 권리/IP 정리, 출고 패키지다.
- 아카이브, 네트워크, Code Studio는 이번 정리 범위에서 삭제/제외 기준으로 본다.
- 데스크톱 버전은 일단 보류한다.
- 백업 폴더가 있으므로 삭제 자체는 가능하되, 신버전에 살릴 수 있는 자산을 먼저 선별한다.

### 2.1 집필 수동 모드 강화

상태: 미진행  
우선순위: P0  
근거 파일:

- `src/components/studio/NovelEditor.tsx`
- `src/components/studio/WritingToolbar.tsx`
- `src/components/studio/tabs/writing/EditModeSection.tsx`

문제:

- 본문 편집기는 Tiptap 기반인데 툴바는 `textareaRef` 기반이다.
- 찾기/바꾸기, 들여쓰기, 장면 전환, 선택 영역 조작이 IDE 중심 에디터와 어긋난다.
- 사용자가 말한 “수동 모드 워드 약함”의 직접 원인이다.

실행안:

1. `WritingToolbar`를 Tiptap command bridge 기반으로 전환.
2. 기존 textarea fallback은 유지하거나 제거 후보로 분리.
3. 찾기/바꾸기, 선택영역 감싸기, 장면 전환, 들여쓰기, 되돌리기 상태를 Tiptap 기준으로 통합.
4. 글자수, 대사 비율, 평균 문장 길이, 리듬 상태를 툴바 하단 상태줄에 배치.

완료 기준:

- 수동 집필 모드에서 textarea 없이 주요 편집 명령이 동작.
- Tiptap selection 기준으로 선택 영역 조작 가능.
- 기존 `NovelEditor` inline rewrite, symbol hover, minimap 기능 회귀 없음.

## 3. 2가지 실행안

### 안 A: 안정형

목표: 신버전 Loreguard의 집필 IDE 완성도 우선.

순서:

1. 수동 집필 Tiptap bridge.
2. 기계결함 감사.
3. 퇴고 보고서/승인 큐.
4. `final_clean` 출고 분리.
5. Work Receipt 2.0.
6. 삭제 후보 리포트.

장점:

- 체감 개선이 빠르다.
- 회귀 위험이 낮다.
- Loreguard 포지션이 분명해진다.

단점:

- 번역/해외 출고 고도화는 뒤로 밀린다.

권장: 현재는 안 A가 우선이다.

### 안 B: 정리형

목표: 삭제/정리와 제품 구조 단순화 우선.

순서:

1. Archive / Network / Code Studio 잔여 참조 전수 탐색.
2. 삭제 후보 리포트.
3. 기능 추출.
4. 삭제 실행.
5. 빌드/테스트 복구.
6. 집필 IDE 보강.

장점:

- 프로젝트가 빨리 가벼워진다.
- 죽은 기능과 미연결 기능이 줄어든다.

단점:

- 사용자가 체감하는 집필 기능 개선이 늦어진다.
- 삭제 중 회귀 대응 시간이 생긴다.

목표: 원고를 제출 가능한 IP 패키지로 만든다.

## 7. 최종 판단

현재 가장 부족한 부분은 “AI 생성”이 아니라 “수동 집필 IDE로서의 밀도”다.

따라서 우선순위는 다음이다.

1. 수동 집필 편집감.
2. 검증 후보와 출고 전 기계결함 감사.
3. 퇴고 승인 흐름.
4. 출고 패키지와 권리/IP 기록.
5. 번역 sign-off.
6. 삭제 정리.

이 순서가 Loreguard를 해줘 시장이 아니라 하라 시장 제품으로 굳히는 데 가장 적합하다.
