# 노아 포착 메모장 정밀 설계

Date: 2026-06-21
Scope: Loreguard Studio 세계관, 캐릭터/아이템, 메인 시나리오, 씬시트, 연출

## 1. 설계 목적

작가가 중앙 작업면에서 자유롭게 말하거나 지시하면, 노아가 옆 메모장에 작품 정보를 실시간으로 포착하고, 작가가 승인한 항목만 각 탭 양식에 반영한다.

목표 감각은 "노아에게 부탁한다"가 아니다. 작가가 현장 감독처럼 지시하고, 노아는 그 말을 작업 가능한 후보로 정리한다.

```text
작가 지시
-> 포착 메모
-> 정리 후보
-> 작가 승인
-> 탭 양식 반영
```

금지 감각:

- 대신 써주는 도구
- 빈 양식을 채우라고 압박하는 도구
- 작가가 도움을 구걸하는 느낌
- 대화 속 농담, 은어, "ㅋㅋㅋ"가 양식에 바로 들어가는 구조

권장 감각:

- 말하면 작업지가 차오른다
- 작가의 말을 기준선으로 접어준다
- 노아는 후보를 정리하고 작가가 결정한다
- 작가가 리드하고 있다는 근자감이 유지된다

## 2. 제품 언어

공개 UI 기본 용어:

| 개념 | 표시 문구 |
|---|---|
| raw capture | 포착 메모 |
| cleaned proposal | 정리 후보 |
| pending approval | 작가 확인 대기 |
| apply | 반영 |
| hold | 보류 |
| discard | 무시 |
| edit before apply | 고쳐서 반영 |
| record | 과정기록 |
| target form | 기준선 / 씬시트 / 연출 메모 |

피할 문구:

- 자동 생성
- AI가 작성
- 추천받기
- 도움 요청
- 문제 해결해줘

권장 버튼:

- 장면 지휘
- 후킹 압력 올리기
- 감정선 조율
- 리듬 재배치
- 대사 후보 정리
- 연출안 정리
- 기준선에 반영
- 씬시트로 옮기기
- 연출 메모로 남기기

## 3. UX 구조

### 3.1 기본 배치

중앙 작업면은 유지한다. 포착 메모장은 오른쪽 도킹 패널로 둔다. 기본은 접힌 상태이며, 포착이 생기면 접힌 바에 배지를 표시한다.

```text
[좌측 도구] [중앙 작업면] [포착 메모장]
```

오른쪽 패널은 세 구역을 갖는다.

```text
1. 포착됨
   작가가 방금 말한 원문 조각

2. 정리 후보
   노아가 양식에 맞게 다듬은 후보

3. 반영 대기
   작가가 확인하면 실제 탭 데이터로 들어갈 항목
```

### 3.2 별도 창 모드

고급 사용자는 메모장을 별도 창으로 분리할 수 있다.

사용 시나리오:

- 듀얼 모니터
- 긴 세계관 구술
- 회차별 씬시트 대량 정리
- 출고 전 권리/IP 근거 확인

MVP에서는 브라우저 팝업 대신 앱 내부 floating panel로 시작한다. 이후 `window.open` 기반 보조창은 별도 단계에서 검토한다.

### 3.3 모바일

모바일에서는 오른쪽 패널 대신 하단 sheet로 연다.

접힌 상태:

```text
포착 3 · 대기 2
```

펼친 상태:

```text
포착 메모 / 정리 후보 / 반영 대기
```

## 4. 처리 단계

### 4.1 1차: 포착 메모

대화 또는 입력에서 작품 관련 조각을 잡아낸다.

예시 원문:

```text
이번 화는 조력자가 수상한 티를 내는데 주인공은 아직 확신 못 함 ㅋㅋ
분위기는 조용하고 찝찝하게. 끝은 물건 하나 발견하는 걸로.
```

포착 메모:

```text
- 조력자가 수상한 티를 낸다.
- 주인공은 아직 확신하지 못한다.
- 분위기는 조용하고 찝찝하다.
- 마지막에 물건 하나를 발견한다.
```

포착 메모는 원문과 가까워도 된다. 이 단계는 아직 양식 반영 전이다.

### 4.2 2차: 정리 후보

포착 메모를 탭 양식에 맞게 다듬는다.

정리 후보:

```text
대상: 씬시트
필드: 감정곡선
값: 안도 -> 위화감 -> 찝찝한 의심
근거: "주인공은 아직 확신 못 함", "조용하고 찝찝하게"
상태: 작가 확인 대기
```

### 4.3 3차: 작가 승인 후 반영

작가가 선택한 후보만 실제 양식에 들어간다.

버튼:

```text
[반영] [고쳐서 반영] [보류] [무시]
```

반영 시 과정기록에는 다음을 남긴다.

- 원문 근거
- 정리 후보
- 반영 대상
- 반영 시각
- 작가가 직접 수정했는지 여부

## 5. 2역할 구조

### 5.1 노아 인터뷰어

역할:

- 작가가 계속 말하게 만든다.
- 말맛과 리액션을 담당한다.
- 양식에 직접 저장하지 않는다.
- 작가에게 지시권이 있다는 톤을 유지한다.

출력:

- 자연어 대화
- 필요 시 후보가 있음을 짧게 안내

금지:

- "제가 작성했습니다"
- "자동으로 완성했습니다"
- 양식 반영 확정 표현

### 5.2 포착 엔진

역할:

- 대화 로그를 읽고 작품 정보만 구조화한다.
- 잡담, 농담, 감탄, 메타 발언을 양식 후보에서 제외한다.
- 확신이 낮으면 `확인 필요`로 둔다.

출력:

```json
{
  "captures": [
    {
      "target": "scene",
      "fieldKey": "emotionCurve",
      "rawEvidence": "주인공은 아직 확신 못 함",
      "cleanValue": "안도 -> 위화감 -> 찝찝한 의심",
      "confidence": 0.82,
      "status": "candidate"
    }
  ]
}
```

금지:

- 원문에 없는 설정 발명
- 잡담을 세계관 설정처럼 확정
- 낮은 확신 항목을 바로 후보로 승격
- 작가 승인 없이 탭 데이터 수정

## 6. 데이터 모델 초안

```ts
export type NoaCaptureTarget =
  | "world"
  | "character"
  | "item"
  | "plot"
  | "scene"
  | "direction";

export type NoaCaptureStatus =
  | "captured"
  | "candidate"
  | "needs_review"
  | "applied"
  | "held"
  | "dismissed";

export interface NoaCaptureMemo {
  id: string;
  projectId: string;
  tabId: string;
  sourceMessageId?: string;
  sourceKind: "author-input" | "noa-reply" | "imported-note";
  rawText: string;
  normalizedText: string;
  target: NoaCaptureTarget;
  fieldKey: string;
  confidence: number;
  status: NoaCaptureStatus;
  createdAt: string;
  updatedAt: string;
  appliedTargetId?: string;
  reason?: string;
}
```

초기 저장 위치:

- 반영 전: 프로젝트별 로컬 포착 원장
- 반영 후: 기존 `StoryConfig` 필드
- 과정기록: 기존 creative log 경로 재사용

`StoryConfig`에 바로 대량 필드를 넣기보다는, 1차는 별도 hook/store로 둔다. 반영된 것만 기존 config에 저장한다.

## 7. 탭별 적용 설계

### 7.1 세계관

대상 필드:

- `corePremise`
- `powerStructure`
- `currentConflict`
- `worldHistory`
- `socialSystem`
- `economy`
- `magicTechSystem`
- `factionRelations`
- `survivalEnvironment`
- `culture`
- `religion`
- `education`
- `lawOrder`
- `taboo`
- `dailyLife`
- `travelComm`
- `truthVsBeliefs`

현재 위험:

- 노아 답변 전체를 선택한 세계관 필드에 병합하는 경로가 있다.
- 캐주얼 답변 전체가 양식에 들어갈 수 있다.

개선:

- `adopt(msg.content)` 직접 병합을 금지한다.
- 노아 답변은 먼저 포착 메모로 들어간다.
- 정리 후보만 선택 필드에 반영한다.

표시 예:

```text
포착됨
"매일 기억이 지워지는 도시"

정리 후보
핵심 전제: 이 도시는 매일 자정 시민의 기억이 초기화된다.

[기준선에 반영] [고쳐서 반영] [보류] [무시]
```

### 7.2 캐릭터/아이템

대상:

- 인물 이름
- 역할
- 욕망
- 결핍
- 비밀
- 말투
- 관계
- 아이템 용도
- 아이템 약점
- 아이템 소유권
- 아이템 권리/IP 메모

현재 기반:

- 노아 대화 도크가 있다.
- JSON 제안 파서가 있다.
- 작가 채택 후 반영 경로가 있다.

개선:

- 새 인물/아이템 생성 후보와 기존 인물 보강 후보를 분리한다.
- "이 말은 기존 인물 A의 결핍으로 보입니다"처럼 대상 추정을 표시한다.

### 7.3 메인 시나리오

대상:

- 7문장 시놉시스
- 3막 구조
- 사건 체인
- 회차별 비트
- 엔딩 잠금
- 반전/떡밥

현재 기반:

- 비트 제안 JSON 파서가 있다.
- 채택 후 회차 비트로 반영하는 경로가 있다.

개선:

- 작가 발화에서 "장기 목표", "중간 반전", "최종 보상"을 포착한다.
- 후보를 바로 비트로 만들지 않고, 먼저 시나리오 메모로 보류할 수 있게 한다.

### 7.4 씬시트

대상:

- 목적
- 갈등
- 공개 정보
- 숨김 정보
- 감정곡선
- 보상감
- 후킹
- 다음 연결

현재 기반:

- `TabDirection`이 씬시트 모드와 연출 모드를 함께 담당한다.
- 구조화 제안과 채택 경로가 있다.

개선:

- 작가 한 줄 지시를 씬시트 8영역 후보로 펼친다.
- "씬 추가"와 "현재 씬 보강"을 분리한다.

표시 예:

```text
작가 지시
이번 화는 조력자를 의심하게 만들고 끝은 물건 발견으로.

정리 후보
목적: 조력자에 대한 첫 의심을 심는다.
갈등: 믿고 싶은 마음과 이상한 단서가 충돌한다.
감정곡선: 안도 -> 위화감 -> 찝찝한 의심
후킹: 사건 현장에서 조력자의 물건이 발견된다.
```

### 7.5 연출

대상:

- 장면 톤
- 연출 의도
- 핵심 대사
- 독자 감정
- 미장센
- 카메라/구도
- 조명
- 사운드
- 액션
- 문장 리듬

현재 기반:

- `노아 연출 제안` 버튼이 있다.
- 작품 톤, 장르, 시놉시스, 세계관, 캐릭터, 기존 씬을 컨텍스트로 넣는다.
- 제안 채택 시 씬으로 반영된다.

개선:

- 버튼을 "도움 요청"이 아니라 "작가 지휘" 톤으로 바꾼다.
- 결과를 대형 답변이 아니라 작은 후보 조각으로 쌓는다.

권장 버튼:

```text
후킹 압력 올리기
감정선 조율
리듬 재배치
대사 후보 정리
연출 메모로 남기기
```

## 8. 실시간 처리 정책

완전 실시간 문자 단위 호출은 비용과 품질 모두 위험하다. 체감은 실시간처럼 보이되, 실제 호출은 문장 단위로 묶는다.

트리거:

- Enter 전송
- 마침표/물음표/느낌표 이후 700ms 정지
- 80자 이상 입력 후 1200ms 정지
- 같은 탭에서 최소 4초 간격

중복 방지:

- `rawText + target + fieldKey` 해시
- 같은 후보가 있으면 새로 만들지 않고 근거만 추가

저확신 처리:

- confidence < 0.55: `확인 필요`
- target 불명확: `메모`
- fieldKey 불명확: `분류 대기`

## 9. 프롬프트 계약

포착 엔진 시스템 계약:

```text
당신은 작가의 말을 양식 후보로 정리하는 포착 엔진입니다.
원문에 없는 설정을 만들지 마십시오.
잡담, 농담, 감탄, 메타 발언은 양식 후보에서 제외하십시오.
작가가 승인하지 않았으므로 반영 완료라고 말하지 마십시오.
출력은 JSON만 사용하십시오.
```

출력 스키마:

```json
{
  "captures": [
    {
      "target": "world|character|item|plot|scene|direction",
      "fieldKey": "string",
      "rawEvidence": "string",
      "cleanValue": "string",
      "confidence": 0.0,
      "reason": "string",
      "needsAuthorCheck": true
    }
  ],
  "ignored": [
    {
      "rawEvidence": "string",
      "reason": "joke|meta|smalltalk|unclear"
    }
  ]
}
```

## 12. 구현 단계

### Phase 1: 공용 원장과 UI 껍데기

- 포착 원장 타입 추가
- 오른쪽 포착 메모장 UI 추가
- 접힌 상태 배지 추가
- 수동 메모 추가/보류/무시 구현

검증:

- 메모 추가 후 새로고침해도 유지
- 보류/무시 상태가 섞이지 않음
- 모바일 sheet로 열림

### Phase 2: 세계관 우선 적용

- 세계관 입력/노아 답변을 포착 메모로 보냄
- 직접 필드 병합 금지
- 후보 승인 시 기존 `WORLD_FIELDS` 필드에 반영

검증:

- `ㅋㅋㅋ` 포함 발화가 원문에는 남지만 기준선에는 정제값만 들어감
- 작가 클릭 전 config가 바뀌지 않음
- 적용 후 `worldFieldEvidence`가 남음

### Phase 3: 캐릭터/아이템, 메인 시나리오 연결

- 기존 JSON 제안 파서 결과를 포착 후보로 변환
- 새 항목/기존 항목 보강 분리
- 후보 수정 후 반영 지원

검증:

- 기존 인물 보강과 신규 인물 생성이 구분됨
- 비트 후보가 바로 회차로 들어가지 않고 승인 대기함

### Phase 4: 씬시트/연출 고도화

- 씬시트 8영역 후보 생성
- 연출 6영역 후보 생성
- "새 씬 추가"와 "선택 씬 보강" 분리
- 지휘 버튼 세트 추가

검증:

- 현재 씬 보강 시 새 씬이 중복 생성되지 않음
- 새 씬 추가 시 sceneId 충돌 없음
- 연출 메모와 씬시트 필드가 섞이지 않음

### Phase 5: 별도 창/고급 작업대

- floating popout
- 폭/위치 저장
- `집중`, `설계`, `연출`, `검토` 프리셋

## 13. 실패 방지 규칙

반드시 지킬 것:

- 작가 승인 전 config 수정 금지
- 낮은 확신 후보 자동 반영 금지
- 원문 근거 없는 후보 생성 금지
- 잡담을 작품 설정으로 저장 금지
- 기존 탭 파서 우회 금지
- 과정기록 없이 반영 금지

좋은 실패:

```text
이 문장은 설정인지 농담인지 애매합니다. 메모로만 둘까요?
```

나쁜 실패:

```text
세계관 금기: 작가는 농담을 즐긴다.
```

## 14. 테스트 계획

단위 테스트:

- 포착 스키마 파싱
- confidence 낮은 후보 분류
- 중복 후보 병합
- 필드별 target routing

컴포넌트 테스트:

- 포착 메모 -> 정리 후보 -> 반영
- 수정 후 반영
- 보류/무시
- 접힌 패널 배지

회귀 테스트:

- 세계관 필드 직접 병합 제거
- 씬 채택 sceneId 충돌 방지
- 연출 후보가 선택 씬 보강과 새 씬 추가를 구분

E2E:

1. 세계관에서 캐주얼 문장 입력
2. 오른쪽 포착 메모 확인
3. 정리 후보 수정
4. 기준선 반영
5. 씬시트로 이동해 같은 세계관 기준이 컨텍스트에 반영되는지 확인

## 15. 성공 기준

사용자 체감:

- 작가가 양식 작성이 아니라 썰 풀이처럼 느낀다.
- 오른쪽 메모장이 알아서 차오르는 느낌이 든다.
- 양식 반영 전 작가가 주도권을 가진다.
- 씬시트/연출 채우는 부담이 줄어든다.

제품 기준:

- 세계관~연출까지 동일한 포착 경험이 이어진다.
- 노아가 대신 쓰는 느낌보다 작가 지휘 느낌이 강하다.
- 과정기록과 출고 패키지로 이어질 근거가 남는다.

## 16. 1차 구현 권장 범위

가장 먼저 할 것:

1. 공용 포착 메모장 UI
2. 세계관 직접 병합 경로 교체
3. 씬시트/연출 후보 카드에 `새 씬 추가` / `선택 씬 보강` 분리
4. 접힌 메모장 배지

이 네 가지가 들어가면 첫 사용감이 가장 크게 바뀐다.

## 17. 20회 시뮬레이션 후 반영 규칙

이 절은 2026-06-21 사용자 관점 20회 시뮬레이션에서 나온 마찰을 반영한 운영 규칙이다.

### 17.1 참견감 제어

포착 메모장은 작가의 작업을 가로막지 않는다.

기본 동작:

- 새 후보가 생겨도 패널을 자동으로 크게 열지 않는다.
- 접힌 상태에서는 `포착 3 · 대기 2`처럼 숫자 배지만 표시한다.
- 같은 입력 흐름에서 후보 카드는 최대 3개까지만 전면 노출한다.
- 나머지는 `더 보기` 안에 쌓는다.

예외:

- 작가가 직접 메모장을 열어 둔 경우
- 후보가 현재 선택 씬/필드와 직접 연결되는 경우
- 낮은 확신 후보가 아니라 명확한 작가 지시인 경우

### 17.2 상태 오해 방지

모든 항목은 다음 상태 중 하나를 반드시 가진다.

```text
포착됨
정리 후보
작가 확인 대기
반영됨
보류
무시
```

금지:

- 작가 클릭 전 `반영됨` 표시
- 노아 응답에 "저장했습니다" 표현
- 후보 카드와 실제 양식 값을 같은 시각 스타일로 표시

### 17.3 원문 근거 표시

정리 후보에는 항상 근거를 둔다.

기본 표시:

```text
정리 후보
후킹: 사건 현장에서 조력자의 물건이 발견된다.

근거
"끝에 물건 하나 발견하는 걸로"
```

원문 근거가 없으면 후보로 올리지 않는다.

### 17.4 말맛 보존

정리 후보는 너무 매끈한 문학 문장보다 작가 발화의 방향을 보존한다.

좋은 후보:

```text
주인공은 조력자를 믿고 싶지만, 단서 때문에 찝찝한 의심을 품는다.
```

나쁜 후보:

```text
운명의 장막 속에서 신뢰와 배신의 비극적 전조가 서서히 피어난다.
```

### 17.5 씬시트/연출 분리

씬시트와 연출은 붙어 있지만 반영 대상은 분리한다.

씬시트 후보:

- 목적
- 갈등
- 공개 정보
- 숨김 정보
- 감정곡선
- 보상감
- 후킹
- 다음 연결

연출 후보:

- 톤
- 연출 의도
- 핵심 대사
- 독자 감정
- 미장센
- 카메라/구도
- 조명
- 사운드
- 액션
- 문장 리듬

작가가 `반영`을 누를 때 다음 중 하나를 선택할 수 있어야 한다.

```text
새 씬 추가
선택 씬 보강
연출 메모로 남기기
보류함으로 이동
```

### 17.6 무AI 모드 보장

연결 키나 크레딧이 없어도 1차 메모는 작동한다.

무AI 모드에서 가능한 것:

- 원문 메모 저장
- 문장 단위 쪼개기
- 현재 탭 기준 임시 분류
- 중복 제거
- 작가 직접 수정 후 반영

노아 정리 기능은 연결 가능할 때만 작동한다.

## 18. 개선 반영 후 10회 재시뮬레이션

전제:

- 포착 메모는 무AI로 즉시 작동
- 노아 정리는 멈춤/전송 후에만 작동
- 후보는 자동 반영하지 않음
- 접힌 패널은 배지만 표시
- 후보마다 원문 근거 표시

| # | 상황 | 개선 후 판단 | 남은 마찰 |
|---:|---|---|---|
| 1 | 초보 작가가 세계관을 농담 섞어 말함 | 원문은 메모에만 남고 정리 후보는 깨끗하게 분리됨 | 농담과 설정의 경계가 애매할 때 확인 질문 필요 |
| 2 | 작가가 긴 세계관을 한 번에 붙여넣음 | 로컬 메모가 먼저 쌓이고 노아 정리는 늦게 붙어도 흐름이 덜 끊김 | 긴 입력은 후보 3개 제한 때문에 일부가 숨겨질 수 있음 |
| 3 | 캐릭터 관계를 즉흥적으로 말함 | 기존 인물 보강 후보로 뜨면 체감이 좋음 | 기존 인물 매칭 실패 시 신규 인물로 오분류될 수 있음 |
| 4 | 시나리오 반전을 말함 | `보류`와 `엔딩/반전` 후보로 남기면 유용함 | 스포일러성 정보는 화면 노출 강도 조절 필요 |
| 5 | 씬시트 한 줄 지시 | 8영역 후보로 펼쳐져 부담이 크게 줄어듦 | 모든 영역을 한 번에 채우면 오히려 생성 냄새가 날 수 있음 |
| 6 | 선택 씬을 보강하려 함 | `새 씬 추가` / `선택 씬 보강` 분리로 중복 생성 위험 감소 | 사용자가 선택 씬을 인지하지 못하면 잘못 반영 가능 |
| 7 | 연출 톤만 지시함 | 톤/리듬/독자 감정 후보가 자연스럽게 잡힘 | 핵심 대사는 자동 후보보다 작가 수정 유도 필요 |
| 8 | 연결 키 없이 첫 사용 | 메모장은 살아 있어 제품 체험이 끊기지 않음 | "정리 후보"가 잠겨 있으면 유료 압박처럼 보일 수 있음 |
| 9 | 모바일에서 사용 | 하단 sheet와 배지 중심이면 방해가 적음 | 하단 sheet가 자주 열리면 글쓰기 공간을 먹음 |
| 10 | 장시간 작업 후 메모 누적 | 과정기록/출고 근거로 가치가 커짐 | 보류함 검색/정리 없으면 메모 더미가 됨 |

### 18.2 재시뮬 결론

개선 후에도 가장 위험한 부분은 "후보가 너무 많이 쌓이는 문제"다. 따라서 1차 구현에서는 많이 잡는 것보다 적게, 정확하게 잡는 편이 낫다.

우선순위는 다음과 같다.

1. 접힌 메모장 배지
2. 원문 근거 표시
3. 작가 승인 전 반영 금지
4. 씬시트/연출 반영 대상 분리
5. 보류함 검색/정리

이 다섯 가지가 지켜지면 작가는 "노아가 대신 쓴다"가 아니라 "내가 말한 걸 작업지가 알아듣는다"에 가깝게 느낀다.
