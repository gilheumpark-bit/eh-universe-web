# P10. Risk Register

## 1. 결론

상용 제품의 가장 큰 위험은 AI 성능 자체보다 신뢰 붕괴다.  
작품을 한 번 망치거나, 출고 가능을 근거 없이 말하거나, 이력을 잃으면 유료 제품으로 유지되기 어렵다.

## 2. 주요 리스크

| ID | 리스크 | 영향 | 대응 |
|---|---|---:|---|
| R-001 | 원고/설정 대량 변경 사고 | High | 미리보기, 분할 적용, receipt |
| R-002 | 탭 전문가 차별성 부족 | High | 탭별 체크리스트와 출력 타입 분리 |
| R-003 | 내부 용어 노출 | Medium | 제품어 매핑 레이어 |
| R-004 | 권리/법률 과장 | High | 검토 필요 표현만 사용 |
| R-005 | 저장/내보내기 실패 | High | preflight, recovery hint |
| R-006 | 팀 작업 충돌 | Medium | receipt, version ref, 권한 정책 |
| R-007 | 비용 과다 | Medium | 깊이/탭별 호출 제한 |
| R-008 | 사용자 피로 | Medium | 일반 타이핑에는 관제 최소화 |
| R-009 | 근거 없는 창작 판단 | Medium | context sufficiency gate |
| R-010 | 데모 가치 불명확 | High | 5분 데모와 샘플 프로젝트 |

## 3. 품질 리스크와 차단 조건

| 차단 조건 | 설명 |
|---|---|
| 대량 수정 직접 적용 | 출시 차단 |
| receipt 없는 중요 변경 | 출시 차단 |
| export 검증 없이 완료 주장 | 출시 차단 |
| 탭 전문가 동일 응답 | 유료 pilot 보류 |
| 복구 힌트 없음 | Studio 패키지 보류 |

## 4. 운영 리스크

| 리스크 | 대응 |
|---|---|
| 고객이 AI 결과를 공식 검토로 오해 | "검토 필요" 문구와 책임 경계 표시 |
| 외부 자료 사용 권리 불명확 | source note와 usage policy |
| 프로젝트 데이터 유출 우려 | 로컬/클라우드 경계 설명 |
| 팀 내 책임 추적 어려움 | receipt와 작업자/시간 기록 |

## 5. 리스크 점수

```text
RiskPriority =
0.40 * Impact
+ 0.30 * Probability
+ 0.20 * DetectabilityInverse
+ 0.10 * RecoveryDifficulty
```

상용 출시 전 우선 차단해야 할 항목:

```text
R-001, R-004, R-005, R-010
```
