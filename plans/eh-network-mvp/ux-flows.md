# EH Network MVP UX Flows

## 1. 첫 진입 홈

홈에서 가장 크게 보여줄 행동은 두 개다.

- 행성부터 둘러보기
- 최신 로그 보기

설명 문구는 짧게 유지한다.

- 행성 등록소 = 세계관 허브
- 관측 로그 = 이야기/기록 스트림
- 정산 결과 = 상태 판정 기록

## 2. 회원 첫 행동

회원 가입 후 첫 CTA:

`행성을 등록하고 첫 관측 로그를 남기세요.`

핵심은 행성 생성과 로그 작성을 분리하지 않는 것이다.

## 3. 행성 등록 + 첫 로그 위저드

### STEP 1. 행성 기본 정보

필수:

- 행성명
- 장르
- 문명 단계
- 운영 목표

선택:

- 분류 코드
- 대표 태그

### STEP 2. 현재 상태

필수:

- 현재 상태
- 한 줄 소개

선택:

- EH 위험도
- 시스템 노출 강도
- 핵심 규칙 3개

### STEP 3. 대표 설정

선택:

- 대표 세력
- 대표 인물
- 초월 대가

### STEP 4. 첫 관측 로그

필수:

- 로그 제목
- 사건 분류
- 본문

선택:

- 지역
- 개입 여부
- EH 영향도
- 후속 상태

## 4. 로그 먼저 쓰는 흐름

관측 로그 작성 화면에 항상 아래 버튼을 둔다.

`신규 행성 만들기`

이 경우 최소 필드만 먼저 받는다.

- 행성명
- 장르
- 목표
- 현재 상태
- 한 줄 소개

즉, `로그 -> 행성 생성`도 허용한다.

## 5. 행성 상세 페이지 구조

### 상단 고정 카드

- 행성명
- 운영자명
- 장르
- 문명 단계
- 운영 목표
- 현재 상태
- EH 위험도
- 한 줄 소개

### 중앙 탭

- 최근 관측 로그
- IF/외전
- 정산 기록
- 피드백

### 우측 위젯

- 대표 로그
- 가장 많이 읽힌 로그
- 최근 정산 결과
- 최근 활동

## 6. 게시글 타입 템플릿

### MVP 1차 오픈

- 운용교범
- 공식해설
- 기술보고
- 관측보고
- 정산보고

### 2차 확장

- 사건기록
- 증언기록
- 회수문서

## 7. 타입별 작성 UX

### observation

- 기본 에디터 중심
- 가장 대중적인 글쓰기 흐름

### incident

- 시간순 섹션 블록
- 사건 배경, 경과, 피해, 후속 상태를 나눠서 입력

### settlement

- 운영자 전용
- 판정, 위험도, 보관 등급 입력을 카드형 필드로 고정

### manual

- 리스트/규정 항목 위주
- 목적, 원칙, 절차, 금지 사항 구획 제공

## 8. 작성 폼 공통 규칙

입력 방식은 세 종류로 나눈다.

- `Select`
  - 문서 유형
  - 발행 주체
  - 위험도
  - 보관 등급
  - 사건 분류
- `Short Text`
  - 문서 번호
  - 대상 행성
  - 기록자
  - 지역
- `Long Text`
  - 본문
  - 비고
  - 개인 판단

## 9. 문서 번호 전략

문서 번호는 자동 생성이 좋다.

예시:

- `NHDC-2026-OBS-001`
- `EHN-2026-SET-014`

권장 포맷:

- `기관코드-연도-타입-일련번호`

## 10. 운영자 흐름

운영자가 할 일은 1차에 좁게 잡는다.

- 승인
- 공식 태그 부여
- 정산보고 생성
- 상태 태그 갱신

운영자 화면은 처음부터 복잡하게 만들지 않는다.

- 승인 대기 글 목록
- 판정 입력 패널
- 태그/상태 변경 패널

## 11. 문서 템플릿 요약

### manual

- 목적
- 기본 원칙
- 운용 절차
- 금지 사항
- 예외 규정

### guide

- 문서 목적
- 핵심 정의
- 분류 체계 설명
- 적용 기준
- 예외 항목
- 실제 예시

### technical

- 개요
- 대상 설명
- 분류 기준
- 핵심 특징
- 비교 분석
- 위험 요소
- 결론

### settlement

- 정산 개요
- 대상 사건 요약
- 현재 상태 판정
- 위험 분석
- 권고 조치
- 후속 관측 필요 사항

### observation

- 관측 개요
- 사건 전개
- 상태 변화
- 관측 결론

## 12. 다국어 범위

MVP 기본 정책:

- UI 라벨은 `KO/EN`
- 사용자 본문은 자유입력
- `JP/CN`은 추후 확장

이유:

- 현재 코드베이스의 JP/CN 번역 품질이 불안정하다.
- 게시판 MVP의 우선순위는 생성 흐름과 데이터 구조 안정화다.
